Loading data in a database

1. Introduction

You have seen in a previous tutorial how to process tweets form a CSV file to compute some statistics. You have filtered undesired data, mapped tweets to the Tweet bean, and validated them using the Bean Validation API.

In this tutorial, you will see how to store these tweets in a database. Since only the business logic changes, you will reuse the same application and register a new processor to load validated tweets in the database.

  • JDK 1.6+
  • Maven
  • Git (Optional)

3. Writing tweets to the database

Here is a quick look of what you have seen in the previous tutorial:

public class Launcher {

    public static void main(String[] args) throws Exception {

        // Build a batch engine
        Engine engine = new EngineBuilder()
            .reader(new FlatFileRecordReader(new File("tweets.csv")))
            .filter(new HeaderRecordFilter())
            .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "name", "message"}))
            .validator(new BeanValidationRecordValidator<Tweet>())
            .processor(new TweetCountProcessor())
            .build();

        // Run the batch engine
        Report report = engine.call();

        // Print the batch execution report
        System.out.println("Total tweets containing #EasyBatch: " + report.getBatchResult());
    }
}

We will change the TweetCountProcessor by a RecordWriter to store tweets in a database instead of counting them.

To keep the tutorial simple, a in-memory database will be used.

Tweets will be inserted in the tweet table.

- Tweet table -
Field Type
id INTEGER
user VARCHAR(32)
message VARCHAR(140)

There are 3 record writers to load data in a relational database:

  • The JdbcRecordWriter: writes records in a database using the JDBC API.
  • The JpaRecordWriter: writes records in a database using the Java Persistence API.
  • The HibernateRecordWriter: writes records in a database using Hibernate.

In this tutorial, we will use the JdbcRecordWriter. To use this writer, we need to supply 3 artifacts:

  • a JDBC connection
  • the SQL query to insert data
  • and a PreparedStatementProvider

First things first, let's create a JDBC connection:

Connection connection = DatabaseUtil.getConnection()

The DatabaseUtil class provides static helper methods to work with the embedded database (not shown here for simplicity's sake).

Next, we need to provide the SQL query to insert tweets in the Tweet table:

String query = "INSERT INTO TWEET values (?, ?, ?)"

Finally, we must implement the PreparedStatementProvider to provide values to the SQL query:

Connection connection = DatabaseUtil.getConnection();
    String query = "INSERT INTO tweet VALUES (?,?,?);";
    JdbcRecordWriter jdbcRecordWriter =
        new JdbcRecordWriter(connection, query, new PreparedStatementProvider() {
            @Override
            public void prepareStatement(PreparedStatement preparedStatement, Object record) {
                Tweet tweet = (Tweet) record;
                preparedStatement.setInt(1, tweet.getId());
                preparedStatement.setString(2, tweet.getUser());
                preparedStatement.setString(3, tweet.getMessage());
            }
});

That's all for the record writer, let put all components together:

public class Launcher {

    public static void main(String[] args) throws Exception {

        //load tweets from tweets.csv
        File tweets = new File("tweets.csv").toURI());
    
        //Start embedded database server
        DatabaseUtil.startEmbeddedDatabase();
    
        //Setup the JDBC writer
        Connection connection = DatabaseUtil.getConnection();
        String query = "INSERT INTO tweet VALUES (?,?,?);";
        JdbcRecordWriter jdbcRecordWriter =
            new JdbcRecordWriter(connection, query, new PreparedStatementProvider() {
                @Override
                public void prepareStatement(PreparedStatement preparedStatement, Object record) {
                        Tweet tweet = (Tweet) record;
                        preparedStatement.setInt(1, tweet.getId());
                        preparedStatement.setString(2, tweet.getUser());
                        preparedStatement.setString(3, tweet.getMessage());
                    }
        });
    
        // Build and run a batch engine
        EngineBuilder.aNewEngine()
            .reader(new FlatFileRecordReader(tweets))
            .filter(new HeaderRecordFilter())
            .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "user", "message"}))
            .writer(jdbcRecordWriter)
            .build().call();
    
        // Dump tweet table to check inserted data
        DatabaseUtil.dumpTweetTable();
    
        // Shutdown embedded database server and delete temporary files
        DatabaseUtil.cleanUpWorkingDirectory();
    }

}

5. Performance considerations

By default, the JDBC connection is in autocommit mode. This will make the JdbcRecordWriter commit a transaction after each record insertion.

For performance sensitive applications, this may not be the wanted behavior. You may want to commit a transaction after the insertion of a chunk of records. This is possible through the JdbcTransactionStepListener. You need to disable the autocommit in the JDBC connection and configure the listener with a commit-interval value:

connection.setAutoCommit(false);
int commitInterval = 10;

EngineBuilder.aNewEngine()
    .reader(new FlatFileRecordReader(tweets))
    .filter(new HeaderRecordFilter())
    .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "user", "message"}))
    .writer(jdbcRecordWriter)
    .recordProcessorEventListener(new JdbcTransactionStepListener(connection, commitInterval))
    .build().call();

In this example, the JdbcTransactionStepListener will commit a transaction after each 10 records.

But there still a last point to take into account. What if we have 25 input records? The JdbcTransactionStepListener will:

  • start and commit a transaction for the first 10 records
  • start and commit a transaction for the next 10 records
  • start a transaction for the remaining 5 records, but since this listener is bound to the record processing step, it will never commit the transaction for the last records (if any)

This is where the JdbcTransactionJobListener comes to play, to commit the last transaction (if any). This is a job listener that will check at the end of the job if there is an active transaction and will commit it. This what will make the last records to be committed to the database:

connection.setAutoCommit(false);
int commitInterval = 10;

EngineBuilder.aNewEngine()
        .reader(new FlatFileRecordReader(tweets))
        .filter(new HeaderRecordFilter())
        .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "user", "message"}))
        .writer(jdbcRecordWriter)
        .recordProcessorEventListener(new JdbcTransactionStepListener(connection, commitInterval))
        .jobEventListener(new JdbcTransactionJobListener(connection, true))
        .build().call();

That's all, we have improved the performance of our application with bulk-writes. Easy Batch provide transaction management support through the JdbcTransactionStepListener and JdbcTransactionJobListener APIs.

6. Summary

In this tutorial, you have learned how to use Easy Batch in a typical enterprise requirement to persist data into a database. Note that you have only provided configuration meta-data and have reused exactly the same components that served for another application. This is how flexible Easy Batch's API is to enforce reuse.

In this tutorial, we have used the JdbcRecordWriter to inserts tweets in the database, but you can find here the same example using the HibernateRecordWriter. There is also another tutorial to load data in a MongoDB instance here.
